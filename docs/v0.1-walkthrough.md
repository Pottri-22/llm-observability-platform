## 3. File-by-file — what's in the scaffold and why

### 3.1 The two stores

**[backend/app/db/postgres.py](../backend/app/db/postgres.py)** · async SQLAlchemy 2.x engine + session factory.
- **What:** Lazily-built engine + sessionmaker exposed via `get_session()` — a FastAPI dependency.
- **Why async:** every other piece of the request path is async (FastAPI handler, ClickHouse insert via `asyncio.to_thread`, Redis idempotency). A sync DB call would park the event loop while bcrypt-verifying, and bcrypt at cost 10 takes ~50ms — that's 50ms of stalled concurrency per request. `asyncpg` keeps the loop free.
- **Defend:** "Why async SQLAlchemy?" → bcrypt verify is on the hot path of every authenticated request; sync would gate concurrency.

**[backend/app/db/clickhouse.py](../backend/app/db/clickhouse.py)** · clickhouse-connect HTTP client wrapper.
- **What:** Singleton sync client + `TRACES_TABLE_DDL` string.
- **Why HTTP not native:** native ClickHouse protocol is faster but tooling is weaker. HTTP at port 8123 lets `curl`/`wget` healthcheck work, lets us debug from the browser, and the throughput delta doesn't matter at 500 traces/sec (we're CPU-bound on the cost calc, not the wire).
- **Why a sync client wrapped in `asyncio.to_thread`:** `clickhouse-connect`'s async support has fewer features and more bugs. The threadpool hop is ~50µs — invisible compared to the ClickHouse insert (~5ms).
- **Defend:** "Why ClickHouse and not Postgres for traces?" → see [ADR-0001](adr/0001-clickhouse-over-postgres-for-traces.md). Short version: Postgres + JSONB hits ~50/sec because of WAL contention; ClickHouse columnar storage sustains 500+/sec on the same 2 vCPU.

### 3.2 The schema

**[backend/alembic/versions/0001_initial_schema.py](../backend/alembic/versions/0001_initial_schema.py)** · Postgres schema · `orgs / projects / api_keys`.
- **What:** Three tables with FK cascade. UUIDs as primary keys.
- **Why UUIDs:** API keys are cross-process objects (the SDK has them, the API stores them, the dashboard shows them). Auto-increment integers leak total counts; UUIDs don't. UUID PKs also let the SDK generate `trace_id`s client-side without a round-trip — the client mints a UUID4, server accepts it.
- **Why FK cascade on delete:** when a project is deleted, all its API keys must die instantly. Soft-delete is for user-facing data (so they can be undeleted), not for credentials.
- **Defend:** "Why store the bcrypt hash, not the key?" → bcrypt is one-way; even an attacker with the database cannot replay an API key without the plaintext. We show plaintext exactly once at creation (in `seed_demo_tenant.py` + future dashboard) and never persist it.

**[backend/app/db/clickhouse.py:60](../backend/app/db/clickhouse.py#L60)** · `TRACES_TABLE_DDL` ·  ClickHouse traces table.
- **PARTITION BY `toYYYYMMDD(ts)`:** one partition per day. ClickHouse can drop entire partitions in O(1) when retention expires (e.g., "delete traces older than 90 days" = drop ~90 partitions).
- **ORDER BY `(project_id, ts, trace_id)`:** ClickHouse sorts data inside each partition by this tuple. Almost every query filters by `project_id` (multi-tenant isolation), then by `ts` (time range), then optionally by `trace_id`. With this ORDER BY, ClickHouse can skip whole granules using primary index sparse markers.
- **Defend:** "Why is `project_id` first in ORDER BY, not `ts`?" → because every query is scoped to one tenant, and ClickHouse's sparse index on the leading column is the most selective. Putting `ts` first would require scanning every project's data within a time range.

### 3.3 The auth path

**[backend/app/auth/api_key.py](../backend/app/auth/api_key.py)** · key gen, prefix extract, bcrypt verify.
- **Format:** `aegis_<env>_<32-char base64url>`. Total length ~40 chars.
- **Why a prefix index:** bcrypt verify is intentionally slow (~50ms at cost 10). If we bcrypt-verified against every row, 1000 keys = 50 seconds per auth. Instead, the first 12 chars (`aegis_live_X`) are stored as a unique-indexed `prefix` column → O(1) lookup → bcrypt-verify against that single candidate.
- **Defend:** "What's the prefix for?" → look-up index. Bcrypt is one-way and slow; we need a fast first-pass filter before the expensive compare.

**[backend/app/api/deps.py](../backend/app/api/deps.py)** · FastAPI `get_tenant` dependency.
- **What:** Parses `Authorization: Bearer <key>`, prefix-lookups, bcrypt-verifies, updates `last_used_at`, returns a `TenantContext` (org_id + project_id + api_key_id).
- **Why a frozen dataclass for `TenantContext`:** immutability prevents handlers from mutating tenant state mid-request and accidentally crossing tenants. The dependency is frozen on entry, type-checked by mypy strict, threaded through every handler.
- **Defend:** "How do you prevent tenant cross-contamination?" → the tenant is resolved exactly once per request from the bcrypt-verified key, immutable from that point. ClickHouse inserts include `org_id` and `project_id` from this object; queries always filter on them; SQL injection is impossible because we never concatenate strings.

### 3.4 The ingest path (the hot path)

**[backend/app/api/v1/traces.py](../backend/app/api/v1/traces.py)** · `POST /v1/traces` and `POST /v1/traces/batch`.
- **What:** Two endpoints, both authenticated. Single trace is for SDK retries; batch is the SDK's primary code path.
- **Why a separate batch endpoint:** SDKs buffer in memory and flush ≤ 500 traces per HTTP call. Forcing one trace per request would 500× the request count and burn TLS handshakes; one batch request amortizes the round-trip.
- **Defend:** "How do you handle SDK retries?" → idempotency_key per trace, 24h Redis TTL. If a trace is retried with the same key, the second attempt returns `duplicate:true` without writing.

**[backend/app/services/trace_writer.py](../backend/app/services/trace_writer.py)** · validates trace + writes to ClickHouse.
- **`_row_for(trace, org_id, project_id)`:** converts a validated `TraceIngest` Pydantic model to a positional tuple for ClickHouse.
- **`write_traces`:** single batch insert via `client.insert("traces", rows, column_names=_COLUMNS)`. Wrapped in `asyncio.to_thread` to avoid blocking the event loop.
- **Why server-side cost computation by default:** the SDK *can* send `cost_usd`, but it's optional. If absent, the server computes from `(model, tokens_in, tokens_out)` against `cost.py`'s catalog. This means a buggy SDK can't lie about cost; the server is the source of truth.
- **Defend:** "Why batch insert and not row-by-row?" → ClickHouse is a columnar store. Single-row inserts create tiny parts that the merge engine has to clean up. Batched inserts (we get up to 500 rows in one request) align with how MergeTree wants data — fewer, larger parts.

**[backend/app/services/cost.py](../backend/app/services/cost.py)** · the cost catalog.
- **What:** `RATES: dict[str, ModelRate]` keyed on model ID. `compute_cost_usd(model, tokens_in, tokens_out)` does the math; `has_known_rate(model)` says whether we know the rate.
- **Why a catalog, not an external pricing API:** providers don't expose stable pricing APIs. The catalog is hard-coded with a comment noting "current as of 2026-Q2; published per-model." Update on rate changes; pin in code so tests are deterministic.
- **Why Groq mapped to zero:** Groq has a free tier we use during sprint. Tracking it as $0 keeps cost charts honest and lets us swap to a paid model in v0.2 without changing schema.
- **Bug we found:** `has_known_rate` originally used `is not ZERO_RATE` (object identity). Groq entries used fresh `ModelRate(0.0, 0.0)` instances, so identity returned `True`. Fixed to value comparison. Lesson: never rely on `is` for value semantics in Python.

### 3.5 Idempotency

**[backend/app/core/idempotency.py](../backend/app/core/idempotency.py)** · Redis-backed `claim_idempotency_key(key)`.
- **What:** `SET NX EX 86400` on `aegis:idem:<key>`. Returns `True` (claimed = fresh) or `False` (already there = duplicate).
- **Why Redis and not Postgres:** Redis `SET NX EX` is one network round-trip and atomic. Postgres equivalent would be `INSERT ... ON CONFLICT DO NOTHING`, which works but every duplicate write hits WAL. Redis sheds the deduplication load off the durable store.
- **Why 24h TTL:** SDKs that retry beyond a day are accepting double-counts already. Longer TTLs grow the keyspace without practical benefit.
- **Defend:** "What if Redis goes down?" → currently the trace 500s out, which is the conservative behavior. v0.4 will add a circuit-breaker that bypasses idempotency on Redis failure (preferring slight duplicate risk over total ingest outage).

### 3.6 Cross-cutting

**[backend/app/main.py](../backend/app/main.py)** · `RequestIdMiddleware` + lifespan + `create_app()`.
- **`RequestIdMiddleware`:** every request gets an `X-Request-ID` header (echoed back); structlog binds it to context vars so every log line in that request prints with the same ID. Critical for tracing one bad request through 8 log lines.
- **`lifespan`:** init Postgres + ClickHouse on boot, dispose on shutdown. FastAPI's lifecycle hook ensures we don't lazily build clients during the first request (which would spike its latency).

**[backend/app/core/exceptions.py](../backend/app/core/exceptions.py)** · `AegisError` hierarchy + handlers.
- **What:** A base exception class with `status_code` + `code` fields. Subclasses for Auth/NotFound/Validation/RateLimit. A FastAPI handler converts them to JSON `{error: {code, message}}`.
- **Why a typed hierarchy and not raw `HTTPException`:** the SDK needs stable error codes (e.g., `auth_error`) to take different actions on different failures. HTTP status alone is too coarse; "401" doesn't distinguish "missing key" from "expired key" from "malformed format."

**[backend/app/core/logging.py](../backend/app/core/logging.py)** · structlog config.
- **Dev:** colored `ConsoleRenderer` for human reading.
- **Prod:** `JSONRenderer` so log shippers (Loki/Cloudwatch) can index every field.
- **Defend:** "Why structlog over stdlib logging?" → structlog produces structured fields (`{event:"trace.batch_written", count:42}`), not parsed strings. Loki/Datadog/Cloudwatch all index structured fields natively.

**[backend/app/schemas/trace.py](../backend/app/schemas/trace.py)** · Pydantic input/output models.
- **`TraceIngest`:** the inbound payload. Field validators cap `metadata` at 10 KB serialized so a misbehaving SDK can't DoS us.
- **`TraceBatch`:** wraps `traces: list[TraceIngest]` with `min_length=1, max_length=500`. The 500 cap is the SDK's flush ceiling — kept in sync.

### 3.7 Operations

**[docker-compose.yml](../docker-compose.yml)** · 4-service local dev stack.
- **One-command dev:** `docker compose up -d` brings up postgres + clickhouse + redis + api.
- **Healthchecks with `depends_on: condition: service_healthy`:** API doesn't start until its dependencies are ready. No more "API is up but DB isn't" five-second window of false positives.
- **Volumes:** named volumes for postgres + clickhouse data so `docker compose down` (without `-v`) preserves them across boots.

**[infra/clickhouse/config.d/listen.xml](../infra/clickhouse/config.d/listen.xml)** · the patch I needed.
- **Why this exists:** see §4.2 below — the Windows/WSL2 IPv6 bug.

**[backend/Dockerfile](../backend/Dockerfile)** · multi-stage build (builder + runtime).
- **Builder:** uses `uv` (fast Rust-based pip alternative) to install deps in a venv.
- **Runtime:** copies the venv from builder into a slim base, runs as non-root `aegis` user (UID 1001), HEALTHCHECK on `/healthz`.
- **Defend:** "Why multi-stage?" → keeps build tools (`uv`, build deps) out of the final image. Runtime image is ~150 MB instead of ~600 MB.

**[backend/scripts/seed_demo_tenant.py](../backend/scripts/seed_demo_tenant.py)** · one-shot tenant seeder.
- **Idempotent on org name "demo":** safe to run multiple times. Always mints a fresh API key (we cannot recover a previous plaintext).
- **Why this is a learning artifact:** future-me building the v0.2 onboarding wizard will copy this flow. The wizard is "this script + a UI."

## 4. The verify session — bugs caught + lessons

The scaffold was 2 days old when I sat down to verify it. Four real bugs surfaced, all worth saving as lessons.

### 4.1 `pyproject.toml` — `readme = "../README.md"` broke the Docker build

**Symptom:** `OSError: Readme file does not exist: ../README.md` during `uv sync` inside the api Docker build.

**Root cause:** the build context is `./backend`, so `../README.md` is *outside the context*. The first `uv sync --no-install-project` succeeds (only deps), but the second `uv sync` triggers an editable install of `aegis-backend` itself, which makes hatchling validate the metadata, which makes it follow the readme path.

**Fix:** removed the `readme` field. The backend isn't published to PyPI — only SDKs will be (in v0.2). Removing the field is correct, not a workaround.

**Defend:** "Why isn't the backend published to PyPI?" → it's a service, not a library. Services ship as container images.

### 4.2 ClickHouse listen on `::` fails on Windows/WSL2 Docker

**Symptom:** ClickHouse container alive but listening on nothing. Healthcheck fails forever with "connection refused."

**Root cause:** the stock `docker_related_config.xml` sets `<listen_host>::</listen_host>` (dual-stack IPv6). Docker Desktop on Windows/WSL2 disables IPv6 in the container netns. ClickHouse's `Listen [::]:8123` attempt fails with "EAI: Address family for hostname not supported" — and it does *not* fall back to IPv4. Server stays alive but never binds anything.

**Fix:** mount [infra/clickhouse/config.d/listen.xml](../infra/clickhouse/config.d/listen.xml) overriding `<listen_host>` to `0.0.0.0`. The override file gets merged into the running config because ClickHouse loads everything in `config.d/` alphabetically; my file comes after the stock one and uses `replace="replace"` to win.

**Defend:** "Why a config override and not a different image?" → the alpine image is otherwise fine; the bug is purely in the default network config. Overriding the minimum surface (one tag) is preferable to switching to a 4× larger non-alpine image.

### 4.3 Healthcheck used `wget --spider` against `localhost`

**Symptom:** even after fixing the listen, the healthcheck still failed despite `wget http://127.0.0.1:8123/ping` returning "Ok." from inside the container.

**Root cause:** the original healthcheck used `wget --spider http://localhost:8123/ping`. BusyBox wget on alpine + Docker's embedded DNS at 127.0.0.11 resolves `localhost` flakily. `--spider` mode also handles 200-with-empty-body inconsistently.

**Fix:** `wget -qO- http://127.0.0.1:8123/ping | grep -q Ok` plus `start_period: 30s` (ClickHouse cold-starts in ~20s on WSL2; the default 0s grace was failing the boot).

**Defend:** "How do you write a reliable container healthcheck?" → (a) use `127.0.0.1` not `localhost`, (b) test the actual response body, not just the connection, (c) give `start_period` enough grace for cold start.

### 4.4 `has_known_rate` used `is not ZERO_RATE` (identity, not value)

**Symptom:** unit test `test_has_known_rate` failed: it asserted `has_known_rate("groq/llama-3.1-8b-instant") is False` but got `True`.

**Root cause:** `has_known_rate` was `model in RATES and RATES[model] is not ZERO_RATE`. The Groq entries in `RATES` used fresh `ModelRate(0.0, 0.0)` instances, not the `ZERO_RATE` constant. `is not` is identity comparison; two `ModelRate(0.0, 0.0)` objects with identical fields are *not* the same object.

**Fix:** changed to value comparison — `rate.input_per_mtok > 0 or rate.output_per_mtok > 0`. The semantic question is "does this model have a non-zero rate?", which is value-shaped.

**Defend:** "When should you use `is` in Python?" → only for sentinel values: `None`, `True`, `False`, `Ellipsis`, and explicit module-level singletons. Never for value equality.

### 4.5 Lessons saved to memory across sessions

- `ClickHouse Docker WSL2 quirk` — relevant if Lattice or Helix add ClickHouse later
- `Secret-handling protocol` — never invite the user to paste keys into chat (we burned one Groq key this session)
- `Groq deprecation cadence` — verify model IDs against `/v1/models` at start of every milestone

## 5. Week 0 · Block 1 — `hello_llm.py`

**Spec (from README §12):** `python hello_llm.py` prints a streamed reply, token count, and computed cost. **Time budget:** 1 hr.

**Result:** [week0/hello_llm.py](../week0/hello_llm.py).

**Output (real run):**
```
>>> Calling Groq (llama-3.3-70b-versatile) with stream=True
------------------------------------------------------------
The main difference between TCP and UDP is that TCP ...
------------------------------------------------------------

Tokens   in:  48
Tokens   out: 65
Tokens   total: 113

Latency  TTFT:            908 ms
Latency  end-to-end:     1050 ms

Cost on Groq llama-3.3-70b-versatile:  $ 0.000000   (free tier)
Cost if gpt-4o-mini:           $ 0.000046
Cost if claude-sonnet-4-6:     $ 0.001119
```

### What I'm supposed to learn from this

1. **The OpenAI SDK is a protocol, not a vendor.** Groq exposes an OpenAI-compatible endpoint at `https://api.groq.com/openai/v1`. Same `from openai import OpenAI`, different `base_url`. This is why Aegis can support OpenAI / Anthropic / Groq / Ollama through one SDK in v0.2 (LiteLLM is a thicker version of this same idea).
2. **Streaming is two things stacked.** The token stream (chunk-by-chunk delta) AND the usage chunk (final chunk with `prompt_tokens` / `completion_tokens`). I asked for usage by passing `stream_options={"include_usage": True}`. Without it, the final chunk is empty and I'd have to estimate tokens from the response text — error-prone.
3. **TTFT vs end-to-end latency are different metrics.** TTFT (time to first token) is what the user perceives — they see *something* at 908ms even though the full reply takes 1050ms. Aegis must capture both because they answer different questions: TTFT = "is the model snappy?" / e2e = "how long until the conversation can continue?".
4. **Cost is a catalog lookup, not a call to the provider.** Providers don't expose stable pricing APIs. Aegis's `cost.py` keeps a hard-coded catalog and the SDK code is `tokens × per-million-rate / 1e6`. Trivial math, but when a provider changes pricing, exactly *one* place in the codebase needs updating.

### Bug we found doing Block 1: Groq deprecated `llama-3.1-70b-versatile`

The README §4 / §6 / §9 / `cost.py` all referenced `llama-3.1-70b-versatile`. It got decommissioned. Replacement: `llama-3.3-70b-versatile` (same 70B class, same free tier).

**Lesson:** Groq deprecates aggressively. Saved as memory `Groq deprecation cadence`. Action item: when v0.2 builds the LiteLLM gateway, add a startup smoke test that calls `client.models.list()` and warns loudly if any catalog model is gone.

## 6. What is **not** in v0.1 yet

Per [README §12](../README.md#12-roadmap-versioned-shipping) v0.1 explicit non-goals:

| Capability | Lands in | Why deferred |
|---|---|---|
| TypeScript SDK | v0.2 | Python-first lets me prove the wire format with one client; TS is a faithful port |
| Multi-provider gateway (LiteLLM) | v0.2 | v0.1 hardcodes one provider; LiteLLM is a 4-hour add when the ingest path is proven |
| Eval engine (Judge/RAGAS/BERTScore/PII) | v0.2 | Evals require traces in the store — can't build evals without first building ingestion |
| Magic-link auth | v0.2 | v0.1 uses paste-key in dashboard (which doesn't exist yet either); proper auth needs Supabase wiring |
| Prompt registry + A/B routing | v0.3 | Higher-order feature on top of stable ingest + eval |
| Drift detection (Celery Beat nightly job) | v0.3 | Needs ≥ 24h of traces to compute baseline |
| Regression detection (1h vs 7d eval) | v0.3 | Needs eval scores on traces — depends on v0.2 |
| CI eval gate | v0.3 | Needs eval engine + CI workflow |
| Helm chart, K8s manifests, RBAC, audit log | v0.4 | Compose-only is fine for live demo at v0.1 ship; K8s is the cloud-domain proof |
| OpenTelemetry self-instrumentation | v0.4 | Polish; not load-bearing for v0.1–v0.3 demos |

**Anti-scope (permanent, will never build):** see [README §13](../README.md#13-whats-not-in-aegis-anti-scope) — fine-tuning workflow, human annotation UI, SOC 2 evidence, air-gapped install, SSO/SAML, mobile app, real-time CRDT trace annotation. Each is a deliberate "no" with a one-line reason.

## 7. Mental model I should have before Block 2

Block 2 (4 hrs) builds the LiteLLM unified gateway. Before starting it, I should be able to answer these without rereading code:

1. **What table does a trace land in, and what's the partition key?** — `traces` MergeTree, partition by `toYYYYMMDD(ts)`.
2. **Which DB stores what?** — Postgres: orgs/projects/api_keys/(future: prompts, audit). ClickHouse: traces (and future evals/drift).
3. **What's the auth flow in 4 steps?** — (1) extract Bearer token, (2) prefix-lookup, (3) bcrypt-verify, (4) update last_used_at.
4. **What's an idempotency key for, and why Redis?** — dedupe SDK retries; Redis SET NX EX is one round-trip atomic.
5. **Why does cost.py exist server-side if the SDK could compute cost?** — server is the source of truth; a buggy SDK can't lie.
6. **Where is the readiness boundary in the stack?** — `/readyz` checks Postgres (`SELECT 1`) and ClickHouse (`SELECT 1`) before saying ok. Redis is checked implicitly only when an idempotency-bearing request arrives.
7. **What runs before/during/after the request?** — `RequestIdMiddleware` binds context, then `get_tenant` resolves auth, then handler executes, then `register_exception_handlers` translates errors to JSON, then middleware echoes the request ID back.

If I can't answer any of these, that's a re-read assignment for tomorrow morning's 6–7 AM block.
